# Project Starter Kit — Orchestrator Guide

This file is the **human-facing operating guide** for driving the AI Software Factory Starter Kit.

It explains:
- when to enter the lifecycle
- which command to start with
- which skills to activate
- what artifacts must exist before progressing
- when to switch to rescue, security, review, or release-readiness flows

The starter kit is not meant to be used as a giant one-shot prompt pack.  
It is meant to be used as a **governed delivery operating system**.

---

## 1. Role of the Orchestrator

The orchestrator is responsible for coordinating the delivery lifecycle.

Its job is to:
- identify the current project situation
- choose the correct entry point
- activate the relevant commands, skills, packs, and templates
- ensure artifacts are produced in the right order
- prevent invalid lifecycle jumps
- ensure evidence exists before claiming progress

The orchestrator does **not** replace:
- the project spec
- the lifecycle commands
- the specialist skills

It coordinates them.

---

## 2. First Decision: What kind of situation is this?

Before doing anything else, classify the project into one of these modes.

### A. New project from scratch
Use when:
- the system does not exist yet
- you have a project spec or can create one
- you need a full lifecycle from requirements to release

**Start with:**
- `commands/START-PROJECT.md`

### B. Weak or incomplete spec
Use when:
- a project exists conceptually
- requirements are incomplete, vague, scattered, or inconsistent
- you need to normalize the spec before serious planning/building

**Start with:**
- `commands/SPEC.md`

### C. Existing project that already has code
Use when:
- the codebase already exists
- you want to improve, secure, extend, or review it
- the project was previously generated by AI or manually built

**Usually start with:**
- `commands/RESCUE-PROJECT.md`

### D. Existing project that appears incomplete or broken
Use when:
- visible features are not behaviorally complete
- there are TODOs in in-scope modules
- CRUD is partial
- the system is inconsistent or misleadingly “complete”

**Start with:**
- `commands/RESCUE-PROJECT.md`

### E. Mature project preparing for review or release
Use when:
- the core build exists
- the project is entering hardening, readiness, or release evaluation

**Start with:**
- `commands/REVIEW.md`
- then `commands/SHIP.md` when appropriate

## 3. Lifecycle Overview

The normal lifecycle is:

```text
START-PROJECT → SPEC → PLAN → BUILD → TEST → REVIEW → SHIP
                         ↑
                     RESCUE
```

### Core rule
Do not skip phases casually.

For example:
- do not go from weak spec directly to BUILD
- do not go from rescue assessment directly to SHIP
- do not go to release without review evidence
- do not treat visual UI as feature completion

---

## 4. What to start with

## If the project is new
1. Read `AGENTS.md`
2. Read `commands/START-PROJECT.md`
3. Confirm the project spec under `specs/`
4. Continue with:
   - `SPEC`
   - `PLAN`
   - `BUILD`
   - `TEST`
   - `REVIEW`
   - `SHIP`

## If the project exists already
1. Read `AGENTS.md`
2. Read `commands/RESCUE-PROJECT.md`
3. Assess current state
4. Decide salvage vs rebuild
5. Re-enter lifecycle through:
   - `PLAN`
   - `BUILD`
   - `TEST`
   - `REVIEW`
   - `SHIP`

## If the main issue is requirements quality
1. Read `AGENTS.md`
2. Start with `commands/SPEC.md`
3. Produce:
   - spec gap report
   - readiness score
   - assumptions log
   - ambiguity register
4. Only then move to `PLAN`

## 6. Required artifacts before moving forward

The orchestrator should not allow phase progression without the minimum artifacts.

## Before BUILD
At minimum:
- authoritative project spec
- slice plan
- traceability foundation
- test strategy
- assumptions log when ambiguity exists

## Before TEST
At minimum:
- implemented slices
- updated traceability
- evidence expectations
- selected test strategy
- business-rule focus areas identified

## Before REVIEW
At minimum:
- test outputs
- functional evidence
- stack conformance
- updated traceability
- relevant security artifacts when applicable

## Before SHIP
At minimum:
- review outputs
- readiness assessment
- unresolved blockers clearly listed
- AppSec/PT status when applicable
- deployment/release artifacts when applicable

---

## 7. Invalid lifecycle jumps to block

The orchestrator must block these patterns:

- weak spec → BUILD
- no planning artifacts → BUILD
- incomplete rescue assessment → large rewrite
- no tests/evidence → REVIEW
- no review outputs → SHIP
- unresolved severe security issues → release-ready declaration
- visual UI only → feature marked complete

---

## 8. How to use packs and templates

The orchestrator should also decide when to activate:
- `packs/`
- `adr-packs/`
- `devops-packs/`
- `test-strategy-packs/`

### Examples
- performance, availability, observability concerns → `packs/`
- architecture choices need documenting → `adr-packs/`
- CI/CD, deployment, rollback, environments → `devops-packs/`
- CRUD-heavy or workflow-heavy system → `test-strategy-packs/`

Templates should be instantiated into:
- `plans/`
- `evidence/`
- `reviews/`
- `adrs/`
- `releases/`

Do not leave important work only implied in the command flow.

---

## 9. Security orchestration

When AppSec or PT applies, the orchestrator must ensure they are explicitly included.

### Typical AppSec artifacts
- `evidence/appsec-test-plan.md`
- `evidence/appsec-findings-tracker.md`
- `reviews/appsec-review-scorecard.md`

### Typical PT artifacts
- `releases/penetration-test-readiness.md`
- PT findings log and retest evidence

The orchestrator must ensure security is not treated as optional when the project risk requires it.

---

## 10. Rescue orchestration

For existing incomplete systems, the orchestrator must ensure:
- rescue mode is used first
- current-state gaps are classified
- salvage vs rebuild is decided
- surgical change control is applied
- traceability and tests are updated after recovery

Do not let rescue turn into uncontrolled rewriting.

---

## 11. Simplicity and change discipline

The orchestrator should enforce:
- no speculative abstraction
- no premature flexibility
- no unrelated cleanup during targeted fixes
- no silent assumptions
- verification attached to each important step or slice

This is especially important in V7.7+.

---

## 12. Recommended operator mindset

When using this kit, act like a delivery coordinator, not a prompt improvisor.

Always ask:
- what phase are we in?
- what is the entry point?
- what skills should be active?
- what artifacts must exist?
- what evidence is required before moving on?
- do we need rescue, AppSec, PT, ADR, NFR, or release readiness support?

If you use the starter kit with that mindset, it becomes much more reliable.

---

## 13. Summary

Use this file as the orchestration guide.

- `AGENTS.md` = governance rules
- `ORCHESTRATOR.md` = human-facing operating guide
- `commands/` = lifecycle phases
- `skills/` = specialist execution workflows
- `templates/` = artifact generators
- `specs/` = project source of truth
- `plans/`, `evidence/`, `reviews/`, `releases/` = delivery outputs

The orchestrator exists to make sure the right things happen in the right order.
